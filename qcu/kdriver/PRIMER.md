# Windows 内核驱动入门

> 从零开始理解驱动、硬件、DMA 与 QCU 的关系

---

## 1. Ring 0 vs Ring 3

CPU 有权限级别，Windows 用两个：

```
Ring 3（用户态）— Python、游戏、浏览器都在这
    ↑ 想访问硬件？必须通过系统调用，Windows 审查
Ring 0（内核态）— Windows 自己、驱动程序在这
    ↑ 直接访问任何内存、任何硬件寄存器，无拦截
```

驱动运行在 Ring 0。崩了不是程序报错，是整台电脑蓝屏。

---

## 2. 驱动做的三件事

### 2.1 注册自己

驱动加载后告诉 Windows："我叫 `\Device\QcuKdrv`，有人找我就找这里"。

用户态程序通过 `\\.\QcuKdrv` 这个名字找到驱动，和打开文件一样。

```c
IoCreateDevice(..., L"\\Device\\QcuKdrv", ...);
IoCreateSymbolicLink(L"\\DosDevices\\QcuKdrv", L"\\Device\\QcuKdrv");
```

### 2.2 响应请求（IRP）

Windows 所有 I/O 操作用 IRP（I/O Request Packet）传递。用户态调
`DeviceIoControl`，Windows 打包成 IRP 发给驱动，驱动处理后返回结果。

```
Python: DeviceIoControl(handle, IOCTL_QCU_ALLOC_PHYSICAL, ...)
           ↓ Windows 打包成 IRP
驱动:  IRP_MJ_DEVICE_CONTROL 处理函数被调用
           ↓ MmAllocateContiguousMemory(size)
           ↓ 把地址写回 IRP，完成
Python 拿到返回值
```

### 2.3 清理

驱动卸载时必须自己释放所有分配的内存、删掉设备对象。
不清理 = 内存泄漏挂在内核里，重启才消失。

## 3. kdmapper 做了什么

正常驱动要微软签名才能加载。kdmapper 绕过这个：

```
正常流程：
  你的 .sys → 微软签名验证 → 通过才加载

kdmapper 流程：
  iqvw64e.sys（Intel 正版签名驱动，但有漏洞 CVE-2015-2291）
      ↓ 利用漏洞获得任意内核读写原语
      ↓ 手动把你的 .sys 复制进内核内存（PE 手动映射）
      ↓ 直接跳转执行 CustomDriverEntry()
  你的代码跑起来了，Windows 以为什么都没发生
```

驱动不会出现在设备管理器，不会有注册表记录，重启就消失——开发期完美。

## 4. 驱动为什么能控制硬件

### 4.1 硬件本质上是一块内存（MMIO）

所有硬件，在 CPU 看来，都是地址空间的一段地址：

```
CPU 的 64 位地址空间：
0x0000_0000_0000_0000
        ...
0x0000_7FFF_FFFF_FFFF  ← 用户态内存
        ...
0xFFFF_8000_0000_0000  ← 内核内存
        ...
0xFFFF_FEXX_XXXX_XXXX  ← 硬件寄存器（MMIO）
```

驱动往硬件寄存器的地址写值，硬件就执行对应操作：

```c
// 伪代码：往 GPU 某个寄存器写"开始计算"
*(volatile uint32_t*)0xFFFFFA00_00000080 = 0x1;
// GPU 硬件检测到写操作 → 开始执行 kernel
```

Ring 0 可以直接读写这些地址，Ring 3 不行。这就是 MMIO（Memory-Mapped I/O）。

### 4.2 驱动与硬件的完整交互

```
驱动往寄存器写值
    ↓ 改变了硬件状态机的输入
硬件在下一个时钟周期感知到变化
    ↓ 按照硬件设计的逻辑转移状态
硬件自主执行（CPU 不参与）
    ↓ 完成后触发中断信号
中断控制器通知 CPU
    ↓
驱动的中断处理函数被调用
```

驱动是**触发器**，硬件是**自动机**，中断是**完成通知**。

## 5. 硬件内部怎么工作

### 5.1 大多数硬件里都有小 CPU

```
你以为的显卡：[ 一堆运算单元，纯硬件逻辑 ]

实际的显卡：
  [ 几千个 shader core ]  ← 真的是简单 CPU
  [ DMA 引擎控制器 ]      ← 有自己的小处理器
  [ 固件（ROM 里跑的代码）] ← 启动时就在执行

实际的网卡（Intel/NVIDIA）：
  [ ARM Cortex-M 核心 ]   ← 真的是 ARM CPU
  [ 跑固件：TCP offload、RSS、中断合并 ]

SSD：ARM 核心跑 FTL（Flash Translation Layer）
硬盘：DSP 处理磁头信号
鼠标：8051 单片机
```

### 5.2 没有 CPU 的部分靠状态机

**时钟信号每跳一次，电路根据当前输入和当前状态，转移到下一个状态**。

以 DMA 引擎为例：

```
状态: IDLE
  收到寄存器写（src=A, dst=B, len=1024）→ FETCH

状态: FETCH
  从 src 读 64 字节 → 放入内部 FIFO → WRITE

状态: WRITE
  从 FIFO 取数据 → 写到 dst
  len -= 64，len==0 → DONE，否则 → FETCH

状态: DONE
  触发中断 → IDLE
```

用晶体管直接实现，几纳秒完成一次状态转移。

### 5.3 从晶体管到 CPU

```
晶体管
  → 与/或/非门（基本逻辑）
  → 加法器 / 多路选择器（组合逻辑）
  → 触发器（存 1 个 bit）
  → 寄存器（存多个 bit）
  → 状态机（寄存器 + 组合逻辑 + 时钟）
  → ALU / 流水线
  → 完整处理器
```

CPU 本质也是状态机，只是状态多到可以执行任意指令，所以叫"通用处理器"。

## 6. 声卡与喇叭

### 6.1 喇叭的工作原理

```
线圈 + 磁铁 + 纸盆

电流进来 → 线圈产生磁场 → 和永久磁铁互斥/吸引
         → 纸盆前后振动 → 压缩空气 → 声音

电流大   → 纸盆推得远 → 振幅大 → 声音响
电流变化快 → 纸盆振动快 → 频率高 → 音调高
```

喇叭只认**模拟电信号**（连续变化的电压）。

### 6.2 声卡的工作

电脑里的音频是数字的（PCM，每秒 44100 个采样值，每个 16bit 整数）。
声卡负责转换：

```
RAM 里的 PCM 数据（整数数组）
    ↓
DAC（Digital-to-Analog Converter，数模转换器）
    ↓ 每个采样值变成对应电压，44100次/秒
模拟电信号
    ↓ 功率放大器
喇叭线圈 → 声音
```

反向录音：

```
麦克风（声音→电压）
    ↓
ADC（Analog-to-Digital Converter）
    ↓ 每秒采样44100次，量化成整数
PCM 数据写进 RAM
```

声卡驱动配置 DAC 的采样率、位深，告诉它从哪块内存读数据，DAC 自己按节拍取数播放。

## 7. DMA（Direct Memory Access）

### 7.1 没有 DMA 的世界

播放音频时 CPU 要全程打工：

```python
while True:
    sample = ram[read_pos]     # CPU 从内存读采样值
    dac_register = sample      # CPU 写进声卡寄存器
    read_pos += 1
    sleep(1/44100)             # 等下一个采样时刻
```

44100 次/秒，CPU 什么都干不了。

### 7.2 DMA 的解决方案

DMA 是**专门负责搬运数据的硬件引擎**，独立于 CPU 运行：

```
CPU 对 DMA 说一次：
  "帮我把 ram[0x1000] 开始的 4096 字节
   每隔 1/44100 秒搬一个 到 DAC 寄存器
   搬完了告诉我"

CPU 去干别的事

DMA 引擎自己：
  tick → 读 ram[0x1000] → 写 DAC
  tick → 读 ram[0x1002] → 写 DAC
  ...（不占用 CPU）
  搬完 → 触发中断 → CPU 来补充下一批数据
```

CPU 从"全程打工"变成"偶尔过来看一眼"。

### 7.3 DMA 为什么需要物理地址

CPU 访问内存用**虚拟地址**，经过 MMU 翻译：

```
虚拟地址 0x7FFF_1234
    ↓ MMU 查页表
物理地址 0x4A20_1234  ← 真实的内存芯片位置
```

DMA 引擎没有 MMU，只认**物理地址**，直接上内存总线抢数据。

程序申请的内存虚拟地址连续，但物理上是碎片的：

```
虚拟：0x10000000 ~ 0x10003000（看起来连续）

物理：
  页A 在内存条某处
  页B 在内存条另一处
  页C 在内存条又一处
```

DMA 按虚拟地址搬，数据是乱的。所以必须给 DMA **物理连续**的内存。

### 7.4 CUDA 的 DMA 流程

没有物理连续内存时：

```
numpy array（物理碎片）
    ↓ CPU 复制到 bounce buffer（物理连续）← 多一次 CPU 复制
GPU DMA 引擎从 bounce buffer 传到显存
```

用内核驱动分配物理连续内存后：

```
MmAllocateContiguousMemory → 物理连续内存
numpy 直接住在这里（np.frombuffer，零拷贝）
    ↓ cuMemHostRegister 告诉 CUDA 这块是物理连续的
GPU DMA 引擎直接从这里传到显存（省掉 CPU 复制）
```

## 8. DMA 外挂

### 8.1 传统外挂的问题

```
游戏进程（内存里有敌人坐标、血量）
    ↑
外挂：ReadProcessMemory() 读游戏内存
    ↑
反作弊：有程序在读我的内存 → 封号
```

反作弊的核心：**监视谁在碰我的内存**。

### 8.2 DMA 外挂的思路

在同一台电脑上读内存会被发现，**从电脑外面读**：

```
游戏电脑（PC-A）
┌─────────────────────────────┐
│  游戏进程（内存有坐标数据）   │
│  反作弊程序（监视一切）       │
│  PCIe 总线                  │── PCIe 插槽
└─────────────────────────────┘
                                      ↑
                            FPGA 卡（伪装成网卡）
                                      ↓
                            第二台电脑（PC-B）
                            处理数据，显示透视
```

FPGA 卡插在游戏电脑的 PCIe 插槽，通过 PCIe 总线直接读物理内存，
数据发到第二台电脑处理。

### 8.3 为什么检测不到

```
反作弊能监视的：
  ✓ 进程调用了哪些 API
  ✓ 有没有 ReadProcessMemory
  ✓ 有没有注入的 DLL
  ✓ 有没有可疑驱动

DMA 外挂实际做的：
  ✗ 没有进程          → 进程扫描扫不到
  ✗ 没有 API 调用     → API 监控无记录
  ✗ 没有注入代码      → 内存扫描干净
  ✗ 没有驱动          → 驱动检测通过
```

读内存发生在 PCIe 硬件层，完全绕过操作系统所有监控机制。

### 8.4 常见硬件

- **PCILeech**：开源安全研究工具，原本用于内存取证
- **廉价 FPGA 卡**（Squirrel、75T 等，约 $35）
- **伪装版**：FPGA 烧成网卡固件，插上去看起来是普通网卡

### 8.5 防御反制：IOMMU

```
Intel VT-d / AMD-Vi（IOMMU）

功能：限制每个 PCIe 设备只能访问被允许的内存范围
效果：FPGA 卡只能访问它自己那块内存，
      不能随意读游戏进程的地址
```

实际情况：很多主板默认关闭 IOMMU，反作弊厂商（EAC、BE）开始检测 IOMMU 状态。攻防持续升级。

---

## 9. 总结：同一套机制的不同用途

| 用途 | 谁用 DMA | 读/写什么 | 目的 |
|------|---------|----------|------|
| 音频播放 | 声卡 DMA | RAM → DAC 寄存器 | 播声音 |
| GPU 计算 | GPU DMA | RAM → 显存 | 上传数据 |
| 网络传输 | 网卡 DMA | RAM → 网卡 TX buffer | 发包 |
| QCU 驱动 | GPU DMA | 物理连续 RAM → 显存 | 省 CPU 复制 |
| DMA 外挂 | FPGA DMA | 游戏进程物理内存 → FPGA | 读坐标 |

技术本身是中性的。PCIe DMA 是现代计算机的基础设施，用途取决于使用者。

# MOROZ 技术白皮书

**副标题：前端压缩与后端坍缩的分层式密码恢复研究框架**

---

## 摘要

MOROZ 是一套面向弱口令、模板口令与强先验候选空间的分层式密码恢复研究框架。与传统全空间枚举、单一静态排序或单后端硬扫方法不同，MOROZ 采用“前端压缩 + 后端坍缩”的总体架构：前端通过多源候选建模、优先搜索与收缩统计，将原始候选空间压缩为高价值 frontier；后端则通过坍缩引擎对该工作区执行相位化显影、局部坍缩与结果读出，从而在较低预算下获得更高的候选集中度与更快的目标聚焦速度。

在系统组成上，MOROZ 由 MSCM、K-Warehouse、ISSC、QCU 与 OPU 等模块协同构成。MSCM 负责多源候选建模，K-Warehouse 负责优先搜索与压缩，ISSC 负责收缩质量观测，QCU 作为核心坍缩后端负责高维候选并存、局部坍缩与读出，OPU 则负责运行期治理与动态纠偏。MOROZ 的目标不是将任一单模块神话化，而是将候选建模、搜索压缩与坍缩显影组织为一条可解释、可扩展、可工程化的系统链路。

实验上，MOROZ 在 toy benchmark、pipeline benchmark 与命中型 benchmark 中均表现出明确的体系优势：相较于随机枚举、纯频率排序和全量坍缩后端，采用前端压缩后再调用真实坍缩引擎的 pipeline 结构，能够显著降低运行时间，并在特定模板目标上保持较高命中质量。结果表明，MOROZ 的核心价值不在于“万能解锁”，而在于对可结构化、可压缩问题实施高强度降维打击。

## 1. 背景与问题定义

### 1.1 问题背景

密码恢复问题在工程上长期存在两种极端方法：

第一种是**全空间枚举**。其优点是定义清晰、覆盖完整；缺点是计算成本极高，对高熵空间几乎无工程可行性。

第二种是**单一排序或单一启发式筛选**。其优点是速度快；缺点是容易陷入偏置，无法稳定处理多源信息，也难以在复杂模板空间里维持鲁棒性。

在实践中，许多真实世界的目标并不属于“完全随机密码”，而更接近以下类型：

- 弱口令
- 模板口令
- 含个人习惯的口令
- 含上下文锚点的口令
- 可被局部压缩的候选簇

这类问题既不适合纯暴力硬扫，也不适合只靠一个静态排序器完成。它们更适合一种分层处理方式：**先压缩，再坍缩；先限域，再显影。**

### 1.2 MOROZ 的核心问题意识

MOROZ 建立在一个简单但关键的判断上：

> 真正困难的不是“有没有后端引擎”，而是“如何把问题压缩到适合后端发挥的工作区”。

也就是说，MOROZ 不是把 QCU、评分器、搜索器等模块随意堆叠在一起，而是明确区分两个阶段：

1. **前端阶段**：候选建模、模板限域、压缩、优先搜索
2. **后端阶段**：候选态并存、动力学推进、局部坍缩、结果读出

只有这两个阶段组织正确，后端的坍缩能力才不会被浪费在高熵均匀空间上。

### 1.3 目标问题范围

MOROZ 主要面向以下问题：

- 低熵或中低熵候选空间
- 模板明显的候选结构
- 含有强先验线索的恢复问题
- 可通过多源信息进行建模的问题
- 可通过前端压缩显著缩小工作区的问题

MOROZ **不承诺**对以下情形具备稳定优势：

- 强随机高熵密码
- 无先验、无模板、无上下文的均匀空间
- 无法形成高价值 frontier 的问题

因此，MOROZ 的定位不是“万能破解器”，而是：

> 一套对可结构化问题实施高强度降维打击的研究级恢复框架。

## 2. 系统定位与总设计思想

### 2.1 MOROZ 的系统定位

MOROZ 不是单一算法名，也不是某个单模块的别名。它是一个**总作战体系**，负责组织不同层级模块，使问题从粗大、分散、低利用率的候选空间，逐步收敛为可被坍缩引擎快速显影的高价值工作区。

一句话概括：

> MOROZ 负责把空间压窄，QCU 负责把目标显影。

### 2.2 核心设计思想

MOROZ 的设计遵循以下原则：

#### 原则 1：前后端分层

前端不做坍缩，后端不做全局原始候选枚举。二者通过 frontier 交接，而不是彼此越权。

#### 原则 2：结构优先于蛮力

MOROZ 首先寻找问题中的：

- 模板
- 先验
- 层级来源
- 可聚类候选簇
- 可压缩路径

而不是默认从原始空间直接硬扫。

#### 原则 3：总系统优先于单模块神话

单独夸大任一模块都会导致误判。MOROZ 的性能来自链路组织，而不是来自某个孤立模块的“万能性”。

#### 原则 4：实验分层

MOROZ 明确区分：

- toy
- benchmark
- real

三种模式，以避免将缩配实验结果误当作系统上限，也避免将完整系统的行为简化为单一局部观察。

### 2.3 体系中的“秒感”机制

MOROZ 所谓的“秒感”或极高效率，并不是来自对强密码学本体的神奇摧毁，而是来自两个过程叠加：

1. 前端将候选空间压缩为较窄工作区
2. 后端在该工作区中快速坍缩掉剩余不确定性

因此，MOROZ 快的对象不是“任意原始大空间”，而是：

> 已经被压缩成可显影、可聚焦、可坍缩的问题空间。

## 3. 系统总体架构

### 3.1 分层结构

MOROZ 推荐按四层来理解：

1. **Source / Prior Layer**
2. **Search / Compression Layer**
3. **Collapse / Readout Layer**
4. **Orchestration / Interface Layer**

对应到当前系统，分别是：

- Source / Prior Layer
  - MSCM
  - whois / 常用词 / 个人线索 / 上下文词源 / 模板定义器

- Search / Compression Layer
  - K-Warehouse
  - ISSC（观测与统计部分）
  - frontier builder

- Collapse / Readout Layer
  - QCU / IQPU
  - OPU

- Orchestration / Interface Layer
  - MOROZ orchestrator
  - CLI / TUI / dashboard
  - benchmark runner
  - Agent adapter
  - export / report / logging

### 3.2 总数据流

标准推荐数据流如下：

```text
Sources
  -> MSCM
  -> K-Warehouse
  -> ISSC(pre-collapse stats)
  -> Frontier
  -> QCU
  -> OPU governance
  -> Readout
  -> MOROZ aggregator
  -> UI / Report
```

### 3.3 为什么要引入 frontier

frontier 是 MOROZ 体系的关键中介。它的意义不在于“简单截 top-k”，而在于提供一个满足以下条件的工作区：

- 候选浓度更高
- 模板噪声更低
- 后端预算更可控
- 更容易形成局部 attractor
- 更适合执行相位化坍缩

没有 frontier，后端容易面对一个过宽、过噪、过平的空间；有了 frontier，后端才更像是在“显影高价值候选簇”，而不是在“大海捞针”。

## 4. 前端压缩层

### 4.1 MSCM：多源候选建模器

MSCM 的职责是把不同来源的信息转成统一的候选建模空间。它负责：

- 组织 source layer
- 汇总 token / phrase / pattern / prior
- 计算静态特征分
- 输出结构化候选对象

MSCM 的输入可以包括：

- 常用词
- 个人词
- domain 线索
- context 线索
- 模板规则
- 外部词源
- 先验权重

MSCM 的输出不是最终答案，而是前端可搜索、可评分、可压缩的候选表达。

### 4.2 K-Warehouse：压缩与优先搜索引擎

K-Warehouse 是 MOROZ 前端的核心压缩器。它的主要能力包括：

- 优先队列扩展
- prefix gate
- full gate
- structure_valid 检查
- 上界估计
- 搜索预算控制
- top-k frontier 构建

K-Warehouse 的价值不是“全覆盖”，而是：

> 在有限预算下，用更高的候选密度逼近高价值工作区。

### 4.3 ISSC：收缩统计与观测层

ISSC 只负责“看”，不负责“打”。

它主要提供：

- entropy
- top-q coverage
- retention ratio
- throughput
- collapse stats
- frontier 质量评估

ISSC 的意义在于给 MOROZ 提供一套可解释的观测语言，使前端压缩结果不只是“好像不错”，而是能被统计描述、被 benchmark 比较。

---

## 5. 后端坍缩层

### 5.1 QCU：坍缩后端 / 动力学显影引擎

QCU 不是普通评分器，也不是单纯 reranker。它的核心职责包括：

- 高维候选态并存
- 相位调制
- 动力学推进
- 局部坍缩
- 读出
- 回写候选显影结果

它最适合处理的是：

- 已经压缩过的 frontier
- 存在明显候选簇的工作区
- 可被相位化区分的候选集

QCU 的强项不是接管整个原始问题，而是：

> 对窄空间里的剩余不确定性进行快速坍缩。

### 5.2 OPU：运行治理核

OPU 不负责候选生成，也不负责替代 QCU 核心。它是运行中的治理与纠偏模块，负责：

- tighten / relax
- suppress / boost
- 资源治理
- 质量治理
- 早停判断
- 噪声抑制

一句话：

> QCU 负责显影，OPU 负责控场。

---

## 6. MOROZ、QCU、OPU 的关系

### 6.1 MOROZ 与 QCU 的关系

MOROZ 是总系统，QCU 是其中的核心坍缩武器之一。它们不是同义词，也不是替代关系。

- MOROZ：总指挥、总链路、总编排
- QCU：高价值工作区中的坍缩引擎

### 6.2 QCU 与 OPU 的关系

QCU 负责动力学推进，OPU 负责 runtime 治理。

- QCU 决定如何显影
- OPU 决定运行过程中要不要变阵、要不要早停、要不要抑制某些分支

### 6.3 为什么不能把单模块神话化

如果把 QCU 神话成“默认全局最优求解器”，会忽略前端压缩的重要性。如果把前端神话成“单靠排序就够了”，则会忽略后端显影对局部目标簇的放大作用。

MOROZ 的优势恰恰来自：

> 前端组织问题，后端完成终结。

---

## 7. 接口与模块边界

### 7.1 接口原则

MOROZ 接口遵循以下原则：

1. MOROZ 只暴露系统级接口
2. QCU / OPU 细节不直接泄漏到 MOROZ 外层
3. 前端候选与后端坍缩解耦
4. toy / benchmark / real 模式统一语义
5. 请求对象稳定，后端实现可替换

### 7.2 核心对象

MOROZ 与后端交互时，建议使用三类核心对象：

- `FrontierCandidate`
- `CollapseRequest`
- `CollapseResult`

对应语义分别是：

- 前端候选表达
- 任务投递表达
- 后端回写表达

### 7.3 推荐边界

- MSCM 不负责 QCU 物理参数
- K-Warehouse 不负责 QCU 内核
- ISSC 不负责搜索与坍缩
- QCU 不负责全局原始空间建模
- OPU 不负责候选语义定义
- Agent 不得越权篡改核心求解语义

## 8. Benchmark 结果总结

### 8.1 toy benchmark

在 toy cat benchmark 中，不同方法表现出明显的结构偏好差异：

- `random_like`：纯枚举，缺乏结构意识
- `freq_only`：高频偏置明显，容易被少数高 prior token 支配
- `moroz_full`：开始形成多源候选团块
- `toy qcu`：可以改变头部排序，但更像轻量增强器
- `qcu v2`：开始具备模板一致性和重排序能力

这些实验说明，MOROZ 前端与 QCU 后端并不是冗余关系，而是不同层级的结构处理器。

### 8.2 real QCU benchmark

在将真实 QCU 重建源码接入 benchmark 后，可以观察到：

- 单独运行真实 QCU 时，系统容易掉入某些 attractor
- 其偏好与静态前端排序并不一致
- 因此真实 QCU 更像“动力学后端”，而不是“全局原始排序器”

这进一步支持了 MOROZ 的分层设计。

### 8.3 MOROZ -> real QCU pipeline benchmark

在 pipeline benchmark 中，采用：

- MOROZ 前端压缩
- real QCU 后端坍缩

相较于全量真实 QCU 直接处理全空间，运行时间出现了数量级下降，同时对某些模板目标保持了更高的命中质量。这说明 frontier 机制是 MOROZ 体系中非常关键的一环。

### 8.4 命中型 benchmark

在带目标的命中 benchmark 中，MOROZ 对某些三层模板目标表现出明显优势；而若前端未将目标保留在 frontier 内，则后端 QCU 无法补救。

这一结果说明：

> MOROZ 的瓶颈不只是后端能力，更在于前端模板覆盖质量。

## 9. 能力边界与限制

### 9.1 MOROZ 擅长什么

MOROZ 擅长的问题包括：

- 低熵或中低熵候选空间
- 模板明确的问题
- 有个人先验的问题
- 有上下文锚点的问题
- 可通过多源信息进行限域的问题

### 9.2 MOROZ 不承诺什么

MOROZ 不承诺：

- 对高熵随机密码稳定有效
- 对无模板均匀空间天然占优
- 对所有问题都具备“秒解”效果
- 将单模块神话为万能开锁器

### 9.3 “秒感”的正确解释

MOROZ 的“秒感”来自：

- 前端显著熵减
- 候选工作区压缩
- 后端快速坍缩

因此其快感属于：

> 先把问题变成可显影，再迅速显影。

而不是对任意原始空间直接魔法秒开。

## 10. 工程实现路线

### 10.1 语言路线

当前最合理的实现路线是：

- Python：主系统、编排、前端规则、benchmark、接口层
- C++ / Rust：未来的性能热点下沉

原因在于 MOROZ 本质上不仅是数值核，还包括：

- 规则系统
- 模板系统
- 编排系统
- benchmark 系统
- 接口系统

因此先用 Python 定型，再将热点下沉，是最自然的演化方式。

### 10.2 模式路线

MOROZ 推荐维护三种模式：

- `toy`
- `benchmark`
- `real`

并强制隔离：

- toy 只用于快速验证结构
- benchmark 用于方法比较
- real 用于真实系统能力观察

禁止将 toy 或缩配结果冒充为完整系统上限。

## 12. 未来路线图

未来的 MOROZ 可以继续沿以下方向推进：

### 12.1 前端模板扩展
增强对更多模板目标的覆盖能力，降低 frontier 漏目标的概率。

### 12.2 frontier 自适应
根据 ISSC 统计与运行期诊断，自适应调整 frontier 尺寸，而不是固定 top-k。

### 12.3 QCU 参数映射优化
提高 frontier 到物理参数空间的映射质量，减少不必要的 attractor 偏差。

### 12.4 OPU 策略库
扩展 runtime 治理策略，使 OPU 不只是简单阈值控制，而是更成熟的运行治理器。

### 12.5 TUI / dashboard / cluster mode
完善 MOROZ 的操作界面、集群运行模式与报告系统，使其更接近真正的研究设施控制台。

## 结语

MOROZ 的真正价值，不在于把某个单模块吹成万能，也不在于把密码恢复重新包装成神话。它的价值在于，把原本分散的候选建模、搜索压缩、统计观测与坍缩显影组织成一条有边界、有接口、有实验支撑的系统链路。

它不是对原始空间做盲目硬扫，而是先构造高价值工作区，再让后端对剩余不确定性进行快速显影。因此，MOROZ 不是一把“万能钥匙”，而是一套能够对可结构化问题实施高强度降维打击的研究框架。

一句话总结：

> MOROZ 负责把问题压到可处理区，QCU 负责把目标从该区域中显影出来。
